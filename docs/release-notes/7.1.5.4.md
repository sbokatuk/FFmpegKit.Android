## What's changed

Bindings for the **FFmpeg 7.1.5** line, built from exactly the same .NET for Android / .NET MAUI binding as the 8.1.7 release. The managed API, package layout, target frameworks and licensing are identical — only the native FFmpeg build differs.

This is the middle of the three lines the upstream fork publishes, and the smallest step away from 8.x if you need to stay off it for now.
### Version numbering has changed

Package versions are now **`<FFmpeg version>.<binding revision>`**. Previously they used the
FFmpegKit *release* number, which is a different scheme that does not track FFmpeg at all —
FFmpegKit `8.1.7` packages FFmpeg `n8.1.2`, and FFmpegKit `7.1.6` packages FFmpeg `n7.1.5`. The
version you install now names the FFmpeg you actually get.

**This is not a downgrade**, even though the number may look lower than what you have:

| Old | New | Same FFmpeg |
| --- | --- | --- |
| `8.1.7.3` | `8.1.2.4` | n8.1.2 |
| `7.1.6.3` | `7.1.5.4` | n7.1.5 |
| `6.0.3.3` | `6.1.6.4` | n6.1.6 |

The old versions are unlisted. If you pinned one exactly it will keep restoring; a floating range
such as `8.*` moves you onto the new numbering automatically.

### Should you use this line?

Choose it if you have validated against FFmpeg 7.x, or if something in your command set regressed on 8.x and you want a supported step back without going all the way to 6.0. Otherwise prefer the latest release.

**This is not a route to 32-bit support.** The 7.1.6 binaries ship `arm64-v8a` and `x86_64` only, the same as 6.0.3 and 8.1.7, and require `minSdkVersion` 24. No published FFmpegKit build from this fork includes `armeabi-v7a` or `x86`; if you still serve 32-bit devices, none of these packages will work for you.

### Coming from `Xamarin.FFmpegKit.*`?

The same migration applies as for 8.1.7 — the package IDs changed and the old ones are no longer published:

```diff
-<PackageReference Include="Xamarin.FFmpegKit.Video.Android" Version="4.5.1" />
+<PackageReference Include="FFmpegKit.Net.Video.Android" Version="7.1.5.4" />
```

The `Ffmpegkit.Droid` namespace is **unchanged**, so no source edits are needed. Be aware that the old packages carried four ABIs including 32-bit, declared `MIT` when they in fact embedded copyleft binaries, and targeted classic Xamarin `MonoAndroid10`. All three of those change here — see the licensing section below.

### Included fixes

This line carries the same binding-level fixes as the 8.1.7 release:

- The `com.arthenica.smartexception` classes that `FFmpegKitConfig`'s static initialiser requires are embedded, so FFmpeg calls do not fail with `NoClassDefFoundError` on device. Upstream ships them neither in the `.aar` nor in its `.pom`.
- Every advertised target framework carries a real binding assembly, verified by packaging tests and by on-device smoke tests that run actual FFmpeg commands on an emulator.
