## What's changed

### Version numbering has changed

Package versions are now **`<FFmpeg version>.<binding revision>`**. Previously they used the FFmpegKit *release* number, which is a separate scheme that does not track FFmpeg at all — FFmpegKit `8.1.7` packages FFmpeg `n8.1.2`, and FFmpegKit `7.1.6` packages FFmpeg `n7.1.5`. The version you install now names the FFmpeg you actually get.

**This is not a downgrade**, even though the number looks lower than what you may have installed:

| Old | New | Same FFmpeg |
| --- | --- | --- |
| `8.1.7.3` | `8.1.2.4` | n8.1.2 |
| `7.1.6.3` | `7.1.5.4` | n7.1.5 |
| `6.0.3.3` | `6.1.6.4` | n6.1.6 |

The binaries are unchanged — this is the same FFmpegKit 8.1.7 build as `8.1.7.3`, renumbered to say so honestly. The old versions are unlisted. If you pinned one exactly it will keep restoring; a floating range such as `8.*` moves you onto the new numbering automatically.

Every release note from here on states both numbers, read from the shipped `libavcodec.so` rather than from upstream's release prose.

### Contents

FFmpeg **n8.1.2**, from FFmpegKit release `8.1.7`. `arm64-v8a` and `x86_64`, `minSdkVersion` 24. Targets `net8.0-android34.0`, `net9.0-android35.0` and `net10.0-android36.0`.

## Coming from `Xamarin.FFmpegKit.*`?

The package IDs changed and the old ones are no longer published:

```diff
-<PackageReference Include="Xamarin.FFmpegKit.Video.Android" Version="4.5.1" />
+<PackageReference Include="FFmpegKit.Net.Video.Android" Version="8.1.2.4" />
```

The `Ffmpegkit.Droid` namespace is **unchanged**, so no source edits are needed. It deliberately does not follow the package name: a namespace rooted at `FFmpegKit` containing a type also called `FFmpegKit` would make `FFmpegKit.Execute(...)` resolve the namespace instead of the class and fail to compile.

Three things do change, and two of them fail at runtime rather than at compile time:

**32-bit ABIs are gone.** The 4.5.1 packages carried `armeabi-v7a`, `arm64-v8a`, `x86` and `x86_64`. These ship **`arm64-v8a` and `x86_64` only**. An app still serving 32-bit devices — or an x86 emulator image — will install and then fail when the native library loads. No FFmpegKit line published by the current upstream includes 32-bit binaries, so an older track is not a workaround.

**Minimum Android version is now API 24** (Android 7.0), as required by the native `.aar`.

**Licence metadata has been corrected.** The old packages declared `MIT`, which was wrong: they embedded copyleft native binaries. Packages now declare `MIT AND LGPL-3.0-only`, or `MIT AND GPL-3.0-only` for the `-gpl` variants. Your obligations have not changed — the metadata was inaccurate before. If you ship a `-gpl` variant in a closed-source app, review that now; upstream's guidance is to use a non-GPL variant. Each package carries the applicable texts under `licenses/`.

Also note FFmpeg itself moved from 4.5 to 8.1.2 — expect behaviour changes across four major versions in filter syntax, defaults and codec availability.

## Fixes carried by these bindings

- The `com.arthenica.smartexception` classes that `FFmpegKitConfig`'s static initialiser requires are embedded. Upstream ships them neither in the `.aar` nor in its `.pom`, so without this every FFmpeg call fails with `NoClassDefFoundError` on device.
- Every advertised target framework carries a real binding assembly. An unpinned Android API level previously produced a valid-but-empty assembly that installed cleanly and exposed nothing.
- `Task`-based `ExecuteAsync` with cancellation, `Action` overloads for the log and statistics callbacks, and managed enum conversions — so a transcode can simply be awaited instead of wrapped in `Task.Run`.
