## What's changed

Bindings for the **FFmpegKit 6.0.3** line, built from exactly the same .NET for Android / .NET MAUI binding as the 8.1.7 release. The managed API, package layout, target frameworks and licensing are identical — only the native FFmpeg build differs.

This exists so you can pin to an older FFmpeg line without giving up the modernised packaging.

### Version scheme

`6.0.3.1` is FFmpegKit **`6.0.3`** plus binding revision **`1`**. The first three components always name the native FFmpegKit build inside the package; the fourth belongs to this repository and increments when the bindings or packaging change while the native library stays put.

### Should you use this line?

Choose it if your FFmpeg commands are validated against 6.x and you would rather not absorb two major versions of upstream behaviour change at the same time as the packaging change. Otherwise prefer the latest release — it gets the newer codecs and upstream fixes.

**This is not a route to 32-bit support.** The 6.0.3 binaries ship `arm64-v8a` and `x86_64` only, the same as 7.1.6 and 8.1.7, and require `minSdkVersion` 24. No published FFmpegKit build from this fork includes `armeabi-v7a` or `x86`; if you still serve 32-bit devices, none of these packages will work for you.

### Coming from `Xamarin.FFmpegKit.*`?

The same migration applies as for 8.1.7 — the package IDs changed and the old ones are no longer published:

```diff
-<PackageReference Include="Xamarin.FFmpegKit.Video.Android" Version="4.5.1" />
+<PackageReference Include="FFmpegKit.Net.Video.Android" Version="6.0.3.1" />
```

The `Ffmpegkit.Droid` namespace is **unchanged**, so no source edits are needed. Be aware that the old packages carried four ABIs including 32-bit, declared `MIT` when they in fact embedded copyleft binaries, and targeted classic Xamarin `MonoAndroid10`. All three of those change here — see the licensing section below.

### Included fixes

This line carries the same binding-level fixes as the 8.1.7 release:

- The `com.arthenica.smartexception` classes that `FFmpegKitConfig`'s static initialiser requires are embedded, so FFmpeg calls do not fail with `NoClassDefFoundError` on device. Upstream ships them neither in the `.aar` nor in its `.pom`.
- Every advertised target framework carries a real binding assembly, verified by packaging tests and by on-device smoke tests that run actual FFmpeg commands on an emulator.
